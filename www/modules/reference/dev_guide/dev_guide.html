Reference Pages & You{ A Style Guide }

We need you guys to churn out more reference pages for Pinwheel, our new digital datebook product. This is the style guide you will use to build digital versions of reference pages. To keep things simple, you will work on a gutted version of BRC (the first incarnation of Pinwheel) that only contains the structure and resources we need to build the reference pages. These will be added to the main project as they are completed and approved. Many changes have been made to these pages.

The Rules

While you are encouraged to be creative with the layouts, you must abide by some very important rules:

  1. Follow the styles and conventions laid out in this document
  2. Markup should be clean, consistent and simple
  3. Replicate the print layouts as closely as possible
  4. Apply responsive design best practices
  5. Pay attention to spacing, font-sizes, font-weights, and colors
  6. Use standard html and default foundation markup as much as possible
  7. Learn and have fun!

The Project

For now, we will just be using svn to manage the project on devo. Open up putty and navigate to your home directory yourname/data/ and checkout the reference pages project.

  1. Paste the following code into putty: svn co file:///svn/refpages/trunk refpages
  2. Put a vhost entry in your .conf file. make sure your paths are correct. this project is not a sportcamp.
  3. Place all reference pages in resources/reference_pages/category/
  4. For now just name each document in the form category-name-xxx.html where "x" is the sequential number of the document in its category.
  5. Place all page-specific images in subfolders called images/ and any scripts/plugins in lib/

Commit your changes at the end of the day, and your work will be reviewed the following morning.

The List

We have a spreadsheet of the most popular reference pages provided by the marketing department. It is a shared google doc, and we all have editing priveleges. Make sure it is sorted by UseCount in descending order. Start at the top, pick a page from the master list, and claim it with a mark in the column with your name so we know who is responsible for it. If you choose to work together on a page, mark both columns.

For pages marked 'interactive' just make the page as simple as possible and move on. These pages just need a placeholder and we'll deal with them later.

Your First Page

Get the example PDFs from https://schooldatebooks.com/resources/. There are multiple versions of some files, but for now we will concentrate on the second option under Reference Pages: Large School Reference Pages.

Examine the PDF and think about how best to recreate it in a responsive grid. You should be able to copy and paste the majority of the text content into our html skeleton and go about dividing the content up into blocks, paragraphs, headings, lists, and images. Most pages shouldn't take more than an hour to throw together, and each of you should be able to get 3-4 done a day. Don't worry about extracting images from the pdfs right now, just use placeholder images and write quick comments about what type of image should go there. The images are all black and white and we will probably be replacing a lot of them with colorful modern images.

As always, make sure any files you added or worked on are committed before you go home so everyone has access to the latest versions. Your work will be reviewed and either added to the official project, or sent back to you with notes on design and usability concerns.

Design Notes

You will notice that some reference pages are perfect for adapting to the digital datebook format. Any page that is a single column of content or whose content is broken into columns and blocks will probably be easy to port to the grid. However, some pages have to be reformatted and redesigned to work because they were designed for print. Still others just don't make any sense in the digital world such as the ones with lined areas meant for handwriting. We have to evaluate each reference page design and decide which category it falls into.

If a page doesn't make sense to you, just mark it as such on the list and move on. We will review these pages later and decide whether they are needed.

Any foundation widgets that rely on javascript click/show/hide events will not work out of the box so we should avoid them.

Some pages are formatted in such a way that users will have to scroll down, then back up to read the second column of info. These pages should be reformatted into a single column or otherwise formatted to increase usability on smaller screens like tablets.

Typography

Foundation 3 uses a modular scale to generate typography. That means a great, logical relationship for your copy. You shouldn't have to change the standard typography defaults. Use the default fonts, and use alternate font-weight, italics, and colors to create different looks. If the design would benefit from alternate fonts you might have to generate a custom image because we have to keep page loads fast and light. Remember to stick to text and CSS as much as possible.

Glyphs

Don't forget, you can use icon/glyphs too. We only have a limited set of glyphs available to the brc project right now, but one is being built for use in reference pages as well. In the meantime you can just add a placeholder glyph from the current set. Cheatsheet available here.

Header Styles

Foundation 3 includes a number of different styles and treatments for typographic elements, such as subheaders or small header segments. Every page should start with an h2 tag, and contain a span with a subheading. After that it's really up to you to design the pages using this document as your guide. Don't hesitate to try new things and ask lots of questions.

Paragraphs

In Foundation, paragraphs fall within the modular scale. This includes their line-height, giving the page a feeling of harmony as you scroll.

It's really easy to add emphasis to copy in Foundation. We've implemented some standards like bold and italics, as well as, our own nifty styles, such as <code>, <kbd> and <small>

You can also put text into and <aside> element to give it some hierarchy on the page. Maecenas faucibus mollis interdum. Donec ullamcorper nulla non metus auctor fringilla. Maecenas faucibus mollis interdum. Donec ullamcorper nulla non metus auctor fringilla.

Links

Sometimes there will be a need for links to online resources. Keep it simple. If there is a url on the page, link it up and make note if the link is dead and needs replaced by placing a standard foundation alert label Dead Link next to it.

This is a standard inline paragraph link.

Lists

Lists are helpful for, well, lists of things. Use lists as much as you can to break up content. Have some fun with styling lists in creative ways.

Tables

Tables are necessary for data that is best presented in table-view, but they are notoriously difficult to use in responsive layouts at smaller sizes because of all the scrolling that is required and also the way wide tables bust out of their container.
Examples: Singular Plural
1st Person: I stop. We stop.
2nd Person: You stop. You stop.
3rd Person: He/She/It stops. They stop.

Images

Put images in the 'images' subfolder inside each category folder. Give them descriptive, semantic names. I have set images to fill their container width 100% using background-size: cover; so they will stretch to fill as much of the screen as they need. Use image placeholder service like placehold.it for most images. Don't worry too much about extracting images from the PDFs, as most of them should be replaced with full-color images.
